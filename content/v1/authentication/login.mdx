# Login

The login page is the entry point to the Laplace Platform. It authenticates users with email and password credentials and routes them to the appropriate starting page based on their role.

## Login page

The login page (`/login`) features a clean, centered card with:

- **Laplace logo and name** at the top
- **Language toggle** (top-right corner) to switch between PT-BR and EN-US before logging in
- **Email field** with placeholder "you@company.com"
- **Password field** with masked input
- **Sign In button** with loading state

The page uses the platform's design system — warm neutrals in light mode, neutral grays in dark mode — and adapts to the user's system theme preference.

## Authentication flow

<Steps>
<Step title="Enter credentials">
Type your email address and password. Both fields are required. The email field has `autoComplete="email"` and the password field has `autoComplete="current-password"` for browser autofill support.
</Step>
<Step title="Submit">
Click **Sign In** (or "Entrar" in Portuguese). The button shows a loading spinner while the request is in progress, and the form is disabled to prevent double submission.
</Step>
<Step title="API authentication">
The frontend sends a `POST /v1/auth/login` request with the email and password. The API verifies the credentials against the database, checks that the user is active, and returns:

- An **access token** (JWT, ~15 minute expiry)
- A **refresh token** (longer-lived, stored in localStorage)

</Step>
<Step title="Token storage">
The access token is stored as a browser cookie (`laplace_access_token`) and the refresh token is saved in localStorage (`laplace_refresh_token`). The cookie has a `max-age` derived from the JWT's expiration claim.
</Step>
<Step title="Post-login routing">
The user is redirected based on their role:

| Role | Redirect |
|------|----------|
| **super_admin** | `/select-tenant` — choose which tenant to work with |
| **admin** | `/` — the Digital Twin map (default page) |
| **user** | `/` — the Digital Twin map (default page) |

</Step>
</Steps>

## Error handling

If authentication fails, an error message appears below the sign-in button. Common error scenarios:

| Error | Cause | What to do |
|-------|-------|------------|
| "Failed to sign in" | Incorrect email or password | Double-check your credentials and try again |
| Network error | API is unreachable | Check your internet connection or contact your administrator |
| Account inactive | Your account has been deactivated | Contact your organization's admin |

<Callout type="info">
There is no "forgot password" self-service flow. If you forget your password, contact your organization's admin. They can resend your invite link, which lets you set a new password.
</Callout>

## Token lifecycle

After login, the platform manages tokens automatically:

### Access token

- Stored as a browser cookie (`laplace_access_token`)
- Expires after approximately 15 minutes
- Included automatically in API requests via the Authorization header
- Checked by Next.js middleware on every route navigation

### Refresh token

- Stored in localStorage (`laplace_refresh_token`)
- Longer lifetime than the access token
- Used to silently obtain a new access token before the current one expires
- The refresh happens in the background — users do not see any interruption

### Token expiration detection

The auth context includes an `isTokenExpired()` helper that checks the JWT's `exp` claim with a 30-second buffer. If the access token is about to expire, the context automatically refreshes it using the refresh token before making API calls.

## Signing out

To sign out:

1. Click your **user avatar** at the bottom of the sidebar
2. In the dropdown menu, click **Sign Out** (or "Sair" in Portuguese)

This clears both the access token cookie and the refresh token from localStorage, then redirects you to the login page.

Alternatively, you can access Sign Out from the **Settings** dropdown in the user menu.

## Language on the login page

The login page includes a language toggle that lets you switch between Portuguese and English before signing in. This is useful for:

- New users who prefer a different language from the browser default
- Shared devices where different operators may prefer different languages

The language preference is preserved after login and applies to the entire platform.
