# Authentication

The Laplace Platform uses a **JWT-based authentication system** with a three-tier role hierarchy and multi-tenant data isolation. Every user belongs to exactly one tenant (organization), and access is controlled at both the tenant and user level.

## Architecture overview

The authentication flow involves three layers:

```
Browser (Next.js)  -->  API (Flask on Cloud Run)  -->  Cloud SQL (PostgreSQL)
   |                         |                              |
   | JWT in cookie +         | Validates JWT                | users table
   | localStorage            | Checks permissions           | invite_tokens table
   |                         | Enforces tenant isolation    | tenant_services table
   | AuthContext provides     |                              | user_types table
   | user state to all       |                              | user_type_modules table
   | components              |                              |
```

### Key design decisions

| Decision | Choice | Why |
|----------|--------|-----|
| **Token format** | JWT (JSON Web Token) | Stateless, includes user metadata, verifiable without DB lookup |
| **Token storage** | Access token in cookie + refresh token in localStorage | Cookie for middleware SSR checks; localStorage for silent refresh |
| **Registration** | Invite-only | No self-registration; admin creates users and sends invite links |
| **User-tenant relationship** | 1:1 (one user, one tenant) | Simplifies data isolation and permission model |
| **Password handling** | User sets own password via invite link | No passwords shared over email or chat |

## How it works

<Steps>
<Step title="Admin invites a user">
A tenant admin creates a new user in the Settings panel, specifying their name, email, role, and user type. The system generates a one-time invite token and sends an email with an activation link.
</Step>
<Step title="User sets their password">
The new user clicks the invite link, which takes them to the `/invite` page. After validating the token, they set their password and their account is activated.
</Step>
<Step title="User logs in">
The user navigates to `/login` and enters their email and password. The API verifies credentials and returns a JWT access token (short-lived, ~15 minutes) and a refresh token (long-lived).
</Step>
<Step title="Tokens are stored">
The access token is stored as a browser cookie (`laplace_access_token`) for server-side middleware checks, and the refresh token is saved in localStorage for silent refresh.
</Step>
<Step title="Requests are authenticated">
Every API request includes the access token in the Authorization header. The API validates the JWT signature, extracts the user's identity and tenant, and enforces row-level security.
</Step>
<Step title="Token refresh">
Before the access token expires, the frontend silently refreshes it using the refresh token. The user does not need to log in again unless both tokens expire.
</Step>
</Steps>

## Auth context

The `AuthContext` React context provides authentication state to all platform components. It exposes:

| Property | Type | Description |
|----------|------|-------------|
| `user` | `AuthUser` | Current user object (id, email, name, role, tenant_id, tenant_name) |
| `modules` | `string[]` | List of module keys the user has access to |
| `facilities` | `string[] | null` | Facility codes the user is restricted to (null = all facilities) |
| `isAuthenticated` | `boolean` | Whether the user is logged in |
| `isSuperAdmin` | `boolean` | Whether the user has the super_admin role |
| `isAdmin` | `boolean` | Whether the user has admin or super_admin role |
| `login()` | `function` | Authenticates with email/password |
| `logout()` | `function` | Clears tokens and redirects to login |
| `hasModule()` | `function` | Checks if the user has access to a specific module key |
| `switchTenant()` | `function` | Switches the active tenant (super admins only) |

The sidebar, settings panel, and all protected pages read from this context to determine what to show and what to hide.

<Callout type="info">
The `hasModule()` function is used extensively in the sidebar to filter navigation items. If a user's tenant does not have access to the Loss Prediction service, the entire Loss Prediction section is hidden from the sidebar.
</Callout>

## Security measures

- **JWT signature verification**: All tokens are cryptographically signed and verified on every request
- **Token expiration**: Access tokens expire after ~15 minutes; refresh tokens have a longer lifetime
- **Cookie security**: Access token cookie is set with `SameSite=Lax` and `path=/`
- **Middleware protection**: Next.js middleware checks for a valid access token cookie on all protected routes, redirecting to `/login` if missing
- **Tenant isolation**: Every API query filters by the user's `tenant_id`, preventing cross-tenant data access
