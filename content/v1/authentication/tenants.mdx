# Tenants

The Laplace Platform is a **multi-tenant system**. Each tenant represents a customer organization (logistics operator or carrier). Tenants are fully isolated from each other — they cannot see each other's data, users, or configurations.

## What is a tenant?

A tenant is an organizational boundary in the platform. Each tenant has:

| Property | Description | Example |
|----------|-------------|---------|
| **ID** | Numeric identifier used in database queries | `1` |
| **Slug** | URL-safe identifier used in file paths and references | `ativa-logistica` |
| **Name** | Human-readable display name shown in the UI | `Ativa Logistica` |
| **Modules** | Set of platform modules the tenant has access to (their subscription) | Digital Twin, Loss Prediction, Database |

## Data isolation

Tenant data isolation is enforced at multiple levels:

### Database level

Every data table that contains tenant-specific information includes a `tenant_id` column. Queries are filtered by the authenticated user's tenant, ensuring:

- A user in Tenant A can never query data from Tenant B
- Even direct API calls with a valid JWT are scoped to the token's tenant
- Database-level row security policies provide an additional safety layer

### API level

The Flask API extracts `tenant_id` from the JWT on every authenticated request and uses it as a mandatory filter in all data queries. There is no API endpoint that returns cross-tenant data (except for super admin endpoints).

### Frontend level

The `AuthContext` provides the current user's `tenant_id` and `tenant_name` to all components. The sidebar displays the active tenant name, and all data fetches include the JWT token that scopes the request to the correct tenant.

## Tenant information in the sidebar

The sidebar displays the current tenant at the top:

- **Tenant logo/icon**: The Laplace logo for the internal "Laplace" tenant, or a colored initial circle for other tenants (deterministic color based on the tenant name)
- **Tenant name**: Displayed next to the logo

For super admins, this area is interactive — clicking it opens the **tenant switcher** dropdown.

## Tenant switching (super admins)

Super admins can switch between tenants without logging out:

<Steps>
<Step title="Click the tenant area in the sidebar">
At the top of the sidebar, click on the current tenant name/logo. A dropdown menu appears.
</Step>
<Step title="Select a tenant">
The dropdown lists all tenants in the system with their name and slug. Click on the tenant you want to switch to.
</Step>
<Step title="Context refreshes">
The sidebar shows a loading indicator while the switch is processed. The API issues a new JWT scoped to the selected tenant, and the page reloads with the new tenant's data and module configuration.
</Step>
</Steps>

<Callout type="info">
Tenant switching is only available to super admins. Regular admins and users are permanently bound to their assigned tenant and cannot see or switch to other organizations.
</Callout>

## Tenant selection page

When a super admin logs in, they are redirected to the **tenant selection page** (`/select-tenant`) instead of the main platform. This page shows:

- The Laplace logo and a "Select Tenant" heading
- A list of all tenants, each displayed as a button with:
  - A building icon
  - The tenant name
  - The tenant slug
  - A loading spinner when the selection is being processed

Clicking a tenant button:

1. Calls the `switchTenant()` function from the auth context
2. Obtains a new JWT scoped to the selected tenant
3. Redirects to the main platform (`/`)

## Tenant management

Super admins can manage tenants from the **Settings > Tenants** tab:

### Creating a tenant

Click **Create Tenant** and provide:

- **Name**: The organization's display name (e.g., "Ativa Logistica")
- **Slug**: A URL-safe identifier (e.g., "ativa-logistica")

The slug is used in internal references and file paths. It should be lowercase, hyphenated, and unique.

### Editing a tenant

Click on an existing tenant to edit its name or slug.

### Managing tenant modules

Each tenant has a set of enabled modules that determine their subscription. To manage modules:

1. Find the tenant in the list
2. Click **Edit Modules**
3. Select or deselect modules from the available list
4. Save changes

The available modules follow the platform's module hierarchy:

```
digital_twin
digital_model
  digital_model.units
services
  services.loss_prediction
    services.loss_prediction.table
    services.loss_prediction.collector
    services.loss_prediction.dashboard
    services.loss_prediction.models
  services.control_tower
  services.load_consolidation
database
```

<Callout type="warning">
Changing a tenant's modules immediately affects all users in that tenant. If you disable a module, users who were accessing it will no longer see it in their sidebar on their next page load.
</Callout>

## Tenant scope in the data pipeline

The multi-tenant architecture extends beyond the frontend into the data pipeline:

| Layer | How tenant scoping works |
|-------|--------------------------|
| **Data Lake** | Files are organized by tenant: `tenants/{tenant_code}/{base}/...` |
| **Data Warehouse** | All tables include a `tenant_id` column for row-level filtering |
| **Loss Prediction** | Predictions are scoped to tenant data; the model processes one tenant at a time |
| **Platform API** | JWT-based tenant extraction on every request |
| **Platform Frontend** | AuthContext provides tenant context to all components |

This end-to-end tenant scoping ensures complete data isolation from ingestion through visualization.
