# Architecture

The Laplace Platform is built as a set of modular, event-driven services that collectively form a digital twin of logistics networks. This section covers the system architecture, how data flows from source to visualization, and how each component fits together.

## Component summary

| Component | Technology | Purpose |
|-----------|-----------|---------|
| **Data Lake** | Cloud Run, Google Drive, GCS | Ingests CSV files from Drive, streams to GCS, publishes events |
| **Data Warehouse** | Cloud Run, Cloud SQL | ETL: transforms and loads data from GCS into PostgreSQL |
| **Loss Prediction** | Cloud Run, LightGBM | ML service that scores CTRCs with loss probability |
| **Platform API** | Cloud Run, Flask | REST API with JWT auth, serves data to the frontend |
| **Platform Frontend** | Vercel, Next.js 16 | Web application with map, tables, collector, settings |
| **Cloud SQL** | PostgreSQL | Central database for all structured data |
| **Pub/Sub** | Google Cloud Pub/Sub | Event bus for service-to-service communication |
| **GCS** | Google Cloud Storage | Blob storage for raw files and ML model artifacts |

## Data flow

The platform follows an **event-driven pipeline** architecture:

<Steps>
<Step title="Source data is synced">
The Data Lake service runs on a 5-minute Cloud Scheduler interval. It scans Google Drive for new or updated CSV files, streams them to GCS, and publishes a message to the `data-lake-updates` Pub/Sub topic.
</Step>
<Step title="ETL transforms and loads">
The Data Warehouse service (subscribed to `data-lake-updates`) receives the event, downloads the CSV from GCS, transforms and cleans the data, and loads it into Cloud SQL using a `replace_by_period` strategy (DELETE + INSERT per time period).
</Step>
<Step title="ML service scores data">
After a successful load, the Data Warehouse publishes to `data-warehouse-updates`. The Loss Prediction service picks up the event, identifies unscored CTRCs, runs the LightGBM model, and writes predictions back to Cloud SQL.
</Step>
<Step title="API serves data">
The Platform API reads from Cloud SQL and serves the data via REST endpoints. All requests are authenticated with JWT and scoped to the requesting user's tenant.
</Step>
<Step title="Frontend renders">
The Next.js frontend fetches data from the API and renders it as interactive visualizations — the map, tables, cards, and charts that users interact with.
</Step>
</Steps>

## Event architecture

Services communicate through **Google Cloud Pub/Sub** topics:

| Topic | Publisher | Consumers | Message content |
|-------|-----------|-----------|-----------------|
| `data-lake-updates` | Data Lake | Data Warehouse | Tenant info, base type, GCS paths |
| `data-lake-updates-dead-letter` | Pub/Sub (DLQ) | Manual inspection | Failed messages |
| `data-warehouse-updates` | Data Warehouse | Loss Prediction, future services | Tenant info, table name, period, row count |
| `data-warehouse-updates-dead-letter` | Pub/Sub (DLQ) | Manual inspection | Failed messages |

Each topic has a **dead letter queue** (DLQ) for messages that fail processing after multiple retries. This ensures no data is silently lost.

<Callout type="info">
The event-driven architecture means new services can be added to the pipeline without modifying existing ones. A future "Load Consolidation" service, for example, would simply subscribe to `data-warehouse-updates` and process the same events.
</Callout>

## Deployment model

| Component | Platform | Instances | Concurrency |
|-----------|----------|-----------|-------------|
| Data Lake | Cloud Run | max 10 | 8 threads per instance |
| Data Warehouse | Cloud Run | max 3 | 1 (sequential processing) |
| Loss Prediction | Cloud Run | configurable | 1 |
| Platform API | Cloud Run | auto-scaled | concurrent |
| Platform Frontend | Vercel | edge-distributed | N/A |

All Cloud Run services are in the `southamerica-east1` region. The frontend is deployed globally via Vercel's edge network.

## Security model

Security is enforced at every layer:

| Layer | Mechanism |
|-------|-----------|
| **Frontend routing** | Next.js middleware checks JWT cookie on every route |
| **API authentication** | `@require_auth` decorator validates JWT and extracts user identity |
| **API authorization** | `@require_permission` decorator checks tenant modules and user permissions |
| **Database** | Row-level security filters by `tenant_id` on all queries |
| **Service-to-service** | IAM service accounts with least-privilege permissions |
| **Network** | Cloud Run services are not publicly accessible (invoked via Pub/Sub or IAM-authenticated HTTP) |
