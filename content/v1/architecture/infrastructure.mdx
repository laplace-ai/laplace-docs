# Infrastructure

The Laplace Platform runs on **Google Cloud Platform (GCP)**, with the frontend hosted on **Vercel**. This page covers the infrastructure setup, service configuration, and how the components are wired together.

## GCP project

| Setting | Value |
|---------|-------|
| **Project ID** | `laplace-data-lake` |
| **Region** | `southamerica-east1` (Sao Paulo) |
| **Why this region** | Lowest latency for Brazilian logistics operators; data residency compliance |

All backend services, the database, and storage buckets are in the `southamerica-east1` region.

## Cloud SQL

The platform uses a single managed PostgreSQL instance for all structured data.

| Setting | Value |
|---------|-------|
| **Instance name** | `dw-instance` |
| **Machine type** | `db-g1-small` (1 vCPU, 1.7 GB RAM) |
| **Database** | `data_warehouse` |
| **Engine** | PostgreSQL |

### Key tables

| Table | Owner service | Description |
|-------|--------------|-------------|
| `base_455_cleaned` | Data Warehouse | Cleaned CTRC shipment data from the 455 base |
| `base_930_*` | Data Warehouse | Daily snapshot data from the 930 base |
| `ctrc_loss_predictions` | Loss Prediction | ML model outputs (probability, risk level, confidence) |
| `units` | Platform API | Logistics facility master data |
| `network_links` | Platform API | Route connections between facilities |
| `users` | Platform API | User accounts |
| `invite_tokens` | Platform API | One-time invite tokens for user onboarding |
| `tenant_services` | Platform API | Module subscriptions per tenant |
| `user_types` | Platform API | User type templates |
| `user_type_modules` | Platform API | Module assignments per user type |

### Database users

| User | Purpose |
|------|---------|
| `dw_user` | Data Warehouse service — write access to data tables |
| `api_readonly_user` | Platform API — read-only access for serving data |

### Connection

Backend services connect to Cloud SQL via the **Cloud SQL Auth Proxy** or direct Cloud SQL connections within Cloud Run (using the `/cloudsql/` socket path). For local development, use the proxy:

```bash
cloud-sql-proxy laplace-data-lake:southamerica-east1:dw-instance --port=5433
```

<Callout type="warning">
The `db-g1-small` instance has 1.7 GB RAM. With a 3.4 GB table (`base_455_cleaned` at ~2.2M rows), queries may require heap fetches when the working set exceeds the buffer cache. A database upgrade is planned for scaling.
</Callout>

## Cloud Run services

All backend services run on Cloud Run as serverless containers.

### Data Lake (`cr-data-lake-ingest`)

| Setting | Value |
|---------|-------|
| **Trigger** | Cloud Scheduler (every 5 minutes) |
| **Max instances** | 10 |
| **Concurrency** | Gunicorn 1 worker, 8 threads |
| **Service account** | `sa-drive-reader@laplace-data-lake.iam.gserviceaccount.com` |

The Data Lake orchestrator scans Google Drive for new or updated CSV files, enqueues upload tasks via Cloud Tasks, and workers stream files to GCS in 8MB chunks.

**File patterns synced:**

| Base | Pattern | Frequency |
|------|---------|-----------|
| 455 | `SSW_455_FULL_{MM}-{YY}.csv` | Monthly snapshots, 12-month rolling window |
| 930 | `SSW_930_{DD}-{MM}-{YY}.csv` | Daily snapshots, 30-day rolling window |

**GCS layout:**

```
tenants/{tenant_code}/{base}/{year}/raw/{filename}
```

### Data Warehouse (`cr-data-warehouse`)

| Setting | Value |
|---------|-------|
| **Trigger** | Eventarc push subscription on `data-lake-updates` |
| **Max instances** | 3 |
| **Concurrency** | 1 (sequential processing) |
| **Ack deadline** | 600 seconds |

The Data Warehouse downloads CSVs from GCS, transforms and cleans the data using pandas, and loads it into Cloud SQL via PostgreSQL COPY. It uses a `replace_by_period` strategy: DELETE all rows for a (tenant_id, periodo) pair, then INSERT the new data. This is fast (65-108 seconds per file) and safe for parallel instances.

### Loss Prediction (`cr-loss-prediction`)

| Setting | Value |
|---------|-------|
| **Trigger** | Push subscription on `data-warehouse-updates` (filter: `attributes.base = "455"`) |
| **Concurrency** | 1 |
| **Ack deadline** | 600 seconds |
| **DLQ** | After 15 failed attempts |

The Loss Prediction service:

1. Receives a notification that new 455 data was loaded
2. Queries Cloud SQL for CTRCs without predictions for the active model version (LEFT JOIN exclusion)
3. Loads the LightGBM model from GCS (version defined in `manifest.json`)
4. Scores each CTRC and writes predictions back to Cloud SQL

### Platform API

| Setting | Value |
|---------|-------|
| **Framework** | Flask |
| **Scaling** | Auto-scaled based on traffic |
| **Auth** | JWT validation on every request |

## Pub/Sub topics

The event-driven architecture uses Google Cloud Pub/Sub for service-to-service communication.

| Topic | Publisher | Consumers |
|-------|-----------|-----------|
| `data-lake-updates` | Data Lake | Data Warehouse (Eventarc) |
| `data-lake-updates-dead-letter` | Pub/Sub (DLQ) | Manual inspection |
| `data-warehouse-updates` | Data Warehouse | Loss Prediction, future services |
| `data-warehouse-updates-dead-letter` | Pub/Sub (DLQ) | Manual inspection |

### Message schemas

**data-lake-updates:**

```json
{
  "tenant_key": "000001__ativa-logistica",
  "tenant_slug": "ativa-logistica",
  "tenant_id": 1,
  "base": "455",
  "paths": ["gs://bucket/tenants/.../raw/SSW_455_FULL_02-26.csv"],
  "timestamp": "2026-02-15T10:30:00Z",
  "event_type": "base_updated"
}
```

**data-warehouse-updates:**

```json
{
  "tenant_id": 1,
  "tenant_slug": "ativa-logistica",
  "base": "455",
  "table_name": "base_455_cleaned",
  "periodo": "2026-02",
  "filename": "SSW_455_FULL_02-26.csv",
  "rows_loaded": 150234
}
```

Both topics use message attributes (`base`, `tenant_id`) for subscription filtering, enabling consumers to only receive relevant events.

## Google Cloud Storage

GCS is used for:

| Bucket/Path | Purpose |
|------------|---------|
| Raw data files | CSV files streamed from Google Drive |
| Sync state | `sync_state.json` files tracking Drive sync progress |
| ML model artifacts | Trained LightGBM model files |
| Model manifest | `manifest.json` defining the active model version |

### Sync state tracking

The Data Lake tracks sync progress using `sync_state.json` files in GCS:

```
tenants/{tenant_code}/{base}/sync_state.json
```

Each file tracks `modifiedTime` and `size` per synced file, with statuses: `queued` -> `done`. This enables incremental sync — only new or changed files are uploaded.

## Network architecture

```
User browser
    |
    | HTTPS
    v
Vercel (CDN edge) ── serves Next.js frontend
    |
    | HTTPS (API calls from browser)
    v
Cloud Run (Platform API) ── validates JWT, queries DB
    |
    | Cloud SQL socket
    v
Cloud SQL PostgreSQL ── returns query results
```

For the data pipeline:

```
Cloud Scheduler (cron)
    |
    | HTTP POST
    v
Cloud Run (Data Lake) ── scans Drive, uploads to GCS
    |
    | Pub/Sub message
    v
Cloud Run (Data Warehouse) ── ETL: GCS -> Cloud SQL
    |
    | Pub/Sub message
    v
Cloud Run (Loss Prediction) ── scores CTRCs, writes predictions
```

<Callout type="info">
All Cloud Run services communicate through Pub/Sub, not direct HTTP calls. This decouples services and allows independent scaling, deployment, and failure handling.
</Callout>

## Monitoring and reliability

| Mechanism | Purpose |
|-----------|---------|
| **Dead letter queues** | Capture failed messages after max retry attempts |
| **Ack deadlines** | 600-second deadline prevents premature message re-delivery during long ETL jobs |
| **Idempotent processing** | Data Lake and Warehouse are designed to be safely retried |
| **Sync state files** | Prevent duplicate uploads by tracking what has already been synced |
| **Health checks** | Cloud Run built-in health monitoring and auto-restart |
